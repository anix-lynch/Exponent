- **Resource constraints:** "limited resources", "conflicting priorities", "everything is urgent"
- **Tradeoff questions:** "bug vs feature", "foundational work vs features", "balance X and Y"

### P6 vs P5 Distinction:
- **P6 (Prioritization):** "Which features to build first?" ‚Üí Focus: RANK options (Impact √ó Confidence √ó Ease)
- **P5 (Segmentation):** "Which users should we focus on?" ‚Üí Focus: DEFINE segments (Persona √ó Behavior √ó Value)

### P6 vs P7 Distinction:
- **P6 (Prioritization):** "What to build first?" ‚Üí Focus: RANK options using RICE
- **P6 (Tradeoff Framing):** "How to balance X and Y?" ‚Üí Focus: FRAME tradeoffs (winners/losers/guardrails)

### P6 vs P1 Distinction:
- **P6 (Prioritization):** "Which features to prioritize?" ‚Üí Focus: RANK what to build
- **P1 (Metric Drop):** "Orders down 25%" ‚Üí Focus: DIAGNOSE why metric dropped

### Red Flags (NOT P6):

- "Which users should we focus on?" ‚Üí P5 (Segmentation)
- "How to balance X and Y?" ‚Üí P7 (Tradeoff Framing)

## üìä CONCEPT BUCKET BREAKDOWN

### BUCKET 1: Classic Feature Prioritization
**Questions:** ~15 | **Priority:** üü¢ GREEN (Master this)

**Board Slide Bullets:**
- **What:** "Which features to build first?" or "Prioritize competing features" - core RICE framework
- **Framework:** List Candidates ‚Üí Score (Impact √ó Confidence √ó Ease) ‚Üí Compute RICE ‚Üí Rank ‚Üí Decide + Communicate
- **Impact (1-3):** 3 = massive user/business impact, 2 = meaningful but local, 1 = minor improvement
- **Confidence (0.4-1.0):** 1.0 = strong data/past proof, 0.7 = some data/informed guess, 0.4 = mostly hypothesis
- **Ease (1-3):** 3 = very easy/quick win, 2 = medium effort, 1 = hard/long-term
- **RICE Score:** Impact √ó Confidence √ó Ease ‚Üí Higher = higher priority
- **Decide + Communicate:** What we do first, what we delay, why (tradeoffs)

**Concrete Examples:**
- "Feature A: Impact 3 √ó Confidence 0.9 √ó Ease 3 = RICE 8.1 ‚Üí Build first"
- "Feature B: Impact 2 √ó Confidence 0.6 √ó Ease 1 = RICE 1.2 ‚Üí Defer"
- "Feature C: Impact 2 √ó Confidence 0.7 √ó Ease 2 = RICE 2.8 ‚Üí Schedule next"

**Representative Questions (Do 5 only):**
- Q158: Choose a product and tell me how you would prioritize new features for it.
- Q1138: How do you prioritize competing features?
- Q1139: How do you prioritize features for a product launch?
- Q1323: How would you determine which features to include in a Minimum Viable Product (MVP)?
- Q92: As a YouTube PM, how would you evaluate the suggestion to develop a tool for content creators to generate ideas with scripts automatically?

**‚ù§Ô∏è Reusable Narrative (Base Story - Adapt for Each Question):**

**Framework:** `List Features ‚Üí Score RICE (Impact √ó Confidence √ó Ease) ‚Üí Rank ‚Üí Decide ‚Üí Communicate`

**Memorizable Answer:**

When prioritizing features, I use the RICE framework: Impact √ó Confidence √ó Ease.

**1Ô∏è‚É£ List All Features** ‚Üí All candidate features.

**2Ô∏è‚É£ Score Impact** ‚Üí 1-3: 3 = massive impact, 2 = meaningful but local, 1 = minor.

**3Ô∏è‚É£ Score Confidence** ‚Üí 0.4-1.0: 1.0 = strong data, 0.7 = some data, 0.4 = hypothesis.

**4Ô∏è‚É£ Score Ease** ‚Üí 1-3: 3 = easy/quick win, 2 = medium, 1 = hard.

**5Ô∏è‚É£ Compute RICE** ‚Üí Impact √ó Confidence √ó Ease for each feature.

**6Ô∏è‚É£ Rank** ‚Üí Highest RICE first.

**7Ô∏è‚É£ Decide & Communicate** ‚Üí What to build first (highest RICE), what to delay, why (tradeoffs).

**Key Principle:** RICE = objective prioritization - Impact √ó Confidence √ó Ease.

### BUCKET 3: Resource-Constrained Prioritization

- **Constraints:** Time (timeline cut), Budget (limited budget), People (few engineers), Scope (must deliver X)

### BUCKET 4: Bug vs Feature / Tradeoff Prioritization
**Questions:** ~10 | **Priority:** üü° YELLOW

**Board Slide Bullets:**
- **What:** "Bug vs feature" or "Balance foundational work vs features" - RICE framework for tradeoffs
- **Approach:** List options (bug fixes, features, foundational work) ‚Üí Score RICE ‚Üí Rank ‚Üí Decide tradeoff
- **Bug Fixes:** Impact = user impact (blocking vs minor), Confidence = 1.0 (known issue), Ease = varies
- **Features:** Impact = user/business value, Confidence = varies, Ease = varies
- **Foundational Work:** Impact = future leverage (compound value), Confidence = varies, Ease = usually low
- **Decision:** Highest RICE wins, but consider strategic importance and dependencies

**Concrete Examples:**
- "Bug blocking users: Impact 3 √ó Confidence 1.0 √ó Ease 2 = RICE 6.0 ‚Üí Fix first"
- "Feature requested: Impact 2 √ó Confidence 0.7 √ó Ease 2 = RICE 2.8 ‚Üí Build next"
- "Foundational work: Impact 3 √ó Confidence 1.0 √ó Ease 1 = RICE 3.0 ‚Üí If it unlocks future features"

**Representative Questions (Do 5 only):**
- Q1536: How would you prioritize between fixing a bug or shipping a new feature?

- Q1244: How would you approach a situation where your business users are pushing for a new business requirement and your engineering team has found 3 bugs, which are repetitive in nature?
- Q1144: How do you resolve conflicting product requirements, and who determines which requirement takes priority?
- Q1539: How would you prioritize multiple requests from different teams?

**‚ù§Ô∏è Reusable Narrative (Base Story - Adapt for Each Question):**

**Framework:** `List Options ‚Üí Score RICE ‚Üí Compare ‚Üí Consider Strategic ‚Üí Decide ‚Üí Communicate`

When prioritizing bugs vs features or tradeoffs, I use RICE to objectively compare.

**1Ô∏è‚É£ List All Options** ‚Üí Bug fixes, features, foundational work.

**2Ô∏è‚É£ Score RICE for Each** ‚Üí 
  - **Bugs:** Impact = user impact (blocking = 3, minor = 1), Confidence = 1.0 (known issue), Ease = varies
  - **Features:** Impact = user/business value, Confidence = varies, Ease = varies
  - **Foundational Work:** Impact = future leverage (compound value), Confidence = varies, Ease = usually low

**3Ô∏è‚É£ Compute RICE & Rank** ‚Üí Highest RICE first.

**4Ô∏è‚É£ Consider Strategic** ‚Üí Strategic importance and dependencies.
